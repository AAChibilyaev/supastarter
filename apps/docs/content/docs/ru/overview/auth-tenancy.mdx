AACsearch изначально проектировался как мультиарендный. Каждый ресурс — поисковые индексы, API-ключи, события использования,
пространства знаний — привязан к **организации**. Межорганизационное чтение никогда не допускается.

## Стек аутентификации

Аутентификация предоставляется [Better Auth](https://better-auth.com), настроенным в `packages/auth/auth.ts`.

Включённые функции:

| Функция                           | Примечания                                         |
| --------------------------------- | -------------------------------------------------- |
| Email + пароль                    | Стандартный поток входа                            |
| Magic links                       | Вход по email без пароля                           |
| Passkeys                          | WebAuthn                                           |
| 2FA (TOTP)                        | Одноразовые пароли на основе времени               |
| OAuth (Google, GitHub)            | Социальный вход                                    |
| Организации                       | Рабочие пространства с членством на основе ролей   |
| Организации только по приглашению | Опционально; настраивается                         |
| Панель администратора             | Внутреннее управление пользователями/организациями |
| Имперсонация сессии               | Для поддержки/отладки                              |

## Организации как рабочие пространства

Организация — основная единица рабочего пространства. Пользователи могут входить в несколько организаций и
переключаться между ними в панели управления. Каждый поисковый индекс, API-ключ и аналитическое событие принадлежит
ровно одной организации.

```
User → Member(role) → Organization
                           │
                    SearchIndex(es)
                    SearchApiKey(s)
                    SearchUsageEvent(s)
                    KnowledgeSpace(s)
```

## Контекст сессии в oRPC

Процедуры oRPC вызываются как один из трёх типов:

| Тип процедуры        | Доступный контекст                   | Используется для                          |
| -------------------- | ------------------------------------ | ----------------------------------------- |
| `publicProcedure`    | Сессия не требуется                  | Публичный обработчик поиска, healthcheck  |
| `protectedProcedure` | `context.user`, `context.session`    | Авторизованные операции панели управления |
| `adminProcedure`     | `context.user` + проверка роли admin | Операции только для администраторов       |

Доступ к сессии в серверном компоненте:

```typescript
import { getSession } from "@auth/lib/server";

const session = await getSession();
```

Доступ к сессии в клиентском компоненте:

```typescript
"use client";
import { useSession } from "@auth/hooks/use-session";

const { user, loaded } = useSession();
```

Доступ к активной организации:

```typescript
"use client";
import { useActiveOrganization } from "@organizations/hooks/use-active-organization";

const { activeOrganization, isOrganizationAdmin } = useActiveOrganization();
```

## Изоляция арендатора — Hard Invariant

Каждый поисковый вызов ДОЛЖЕН передавать `tenantId: verified.organizationId`. Это применяется в
`packages/api/modules/search/public-handler.ts`:

```typescript
// Всегда объединяем через AND с фильтром арендатора
const tenantFilter = `organization_id:=${organizationId}`;
const combinedFilter = combineFilters(tenantFilter, callerFilter);
```

Нет кодового пути, который возвращал бы документы разных организаций. При написании новой
процедуры поиска этот инвариант должен явно соблюдаться — он не обеспечивается автоматически.

## Модель безопасности API-ключей

API-ключи — основной механизм аутентификации для внешних вызывающих сторон (модулей CMS и браузерных SDK).

### Типы ключей и префиксы

| Префикс          | Область                  | Используется                           |
| ---------------- | ------------------------ | -------------------------------------- |
| `ss_search_*`    | `search`                 | Браузерный SDK, виджет — только чтение |
| `ss_connector_*` | `connector_write`        | Модули CMS — запись + синхронизация    |
| `ss_scoped_*`    | Сужение от `ss_search_*` | На пользователя, токены с фильтрацией  |

### Гарантии безопасности

- **Хранение только хэша:** `SearchApiKey.hashedKey` хранит только bcrypt-хэш. Открытый текст показывается
  один раз при создании и никогда не логируется, не возвращается при операциях чтения и не дублируется.
- **Ограничение по источнику:** Каждый ключ имеет `allowedOrigins[]`. Запросы с незарегистрированных источников
  отклоняются на уровне API до любого вызова Typesense.
- **Ограничение запросов:** Скользящее окно на ключ через `SearchRateLimitBucket`.
- **Квота:** Квота плана на организацию применяется через middleware `quotaCheck` перед поиском/приёмом данных.
- **Истечение:** Ключи могут иметь `expiresAt`; истёкшие ключи отклоняются и очищаются заданием технического обслуживания.

### Изоляция административного ключа

Административный ключ Typesense **никогда** не покидает сервер. Он живёт в `packages/search/lib/client.ts`,
инициализируется один раз из переменной окружения `TYPESENSE_API_KEY`. Он никогда не возвращается клиенту,
не передаётся модулям CMS и не встраивается в бандл виджета.

Модули CMS получают только токены `ss_connector_*`, которые проходят через Connector API AACsearch,
который затем внутренне использует административный ключ.

## Ограниченные токены

Ограниченные токены сужают разрешения существующего ключа `ss_search_*`. Это stateless JWT, подписанные HMAC
(не хранятся в БД), выпускаемые `issueScopedSearchToken()`.

Типичный случай использования: выпустить токен на пользователя, ограничивающий результаты только товарами этого пользователя.

```typescript
// Сторона сервера: выпуск ограниченного токена для конкретного диапазона цен
const token = await orpc.search.createScopedToken.call({
	organizationId,
	indexSlug: "products",
	scopedFilter: "price:<100",
	expiresInSeconds: 3600,
});
```

`scopedFilter` всегда объединяется через AND с собственными фильтрами вызывающей стороны через `combineFilters()`.
Он может только сужать разрешения — не расширять их. Обход `combineFilters` является нарушением Hard Invariant.

## Список разрешённых источников (Allow-list) по ключу

Каждый ключ `ss_search_*` имеет массив `allowedOrigins[]`. Проходят только запросы с этих источников.

На бесплатном плане разрешены только поддомены `aacsearch.com`. Pro и выше поддерживают пользовательские источники.
Подробности политики по планам смотрите в разделе [Планы и ограничения](/docs/overview/plans-and-limits).

## Ролевой доступ в организациях

| Роль     | Возможности                                                    |
| -------- | -------------------------------------------------------------- |
| `owner`  | Полный доступ, может передать организацию, управлять биллингом |
| `admin`  | Управление индексами, ключами, участниками, настройками        |
| `member` | Просмотр панели управления, выполнение поисков в превью        |

Проверки ролей в процедурах oRPC используют объект `context.session`, предоставляемый Better Auth.
