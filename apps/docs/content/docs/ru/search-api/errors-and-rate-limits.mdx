AACsearch сопоставляет все ошибки на стороне сервера с типизированными кодами ошибок перед возвратом вызывающим сторонам.
Сырые сообщения об ошибках Typesense никогда не пересылаются — это предотвращает утечку информации и
обеспечивает стабильный контракт ошибок.

## Формат ответа с ошибкой

Все ответы с ошибками следуют этому формату:

```json
{
	"error": "error_code",
	"message": "Human-readable description of the error"
}
```

Поле `error` машиночитаемо и стабильно. Поле `message` предназначено для людей
и может изменяться между версиями.

## HTTP-статус коды и коды ошибок

### 4xx — Ошибки клиента (не повторять автоматически)

| HTTP | Код `error`           | Причина                                                         |
| ---- | --------------------- | --------------------------------------------------------------- |
| 400  | `invalid_request`     | Некорректный JSON или отсутствующее обязательное поле           |
| 400  | `invalid_filter`      | Некорректный синтаксис выражения `filterBy`                     |
| 400  | `invalid_sort`        | Некорректное выражение `sortBy`                                 |
| 401  | `unauthorized`        | Отсутствующий, недействительный или истёкший API-ключ или токен |
| 401  | `token_expired`       | Истёк TTL ограниченного токена                                  |
| 403  | `forbidden`           | У ключа нет требуемой области (`search` или `connector_write`)  |
| 403  | `origin_not_allowed`  | Источник запроса отсутствует в `allowedOrigins` ключа           |
| 404  | `index_not_found`     | Указанный `indexSlug` не существует для данной организации      |
| 429  | `rate_limit_exceeded` | Превышен лимит запросов по ключу                                |
| 429  | `quota_exceeded`      | Ежемесячная квота поисковых единиц организации исчерпана        |

### 5xx — Ошибки сервера (возможен повтор с задержкой)

| HTTP | Код `error`           | Причина                                                           |
| ---- | --------------------- | ----------------------------------------------------------------- |
| 502  | `search_failed`       | Ошибка Typesense на стороне сервера (кластер недоступен, таймаут) |
| 502  | `ingest_failed`       | Не удалось поставить документы в очередь буфера приёма данных     |
| 503  | `service_unavailable` | API AACsearch временно недоступен                                 |

## Ограничение запросов

Ограничение запросов применяется по API-ключу с помощью счётчика скользящего окна.

**Лимит по умолчанию:** 60 запросов в минуту на ключ.

При превышении лимита ответ выглядит так:

```
HTTP/1.1 429 Too Many Requests
Content-Type: application/json
Retry-After: 15
X-RateLimit-Limit: 60
X-RateLimit-Remaining: 0
X-RateLimit-Reset: 1717201234

{
  "error": "rate_limit_exceeded",
  "message": "Rate limit of 60 req/min exceeded. Retry after 15 seconds."
}
```

### Заголовки ответа

Эти заголовки включены во все успешные и ответы с ограничением запросов:

| Заголовок               | Описание                                       |
| ----------------------- | ---------------------------------------------- |
| `X-RateLimit-Limit`     | Всего запросов разрешено за окно               |
| `X-RateLimit-Remaining` | Запросов, оставшихся в текущем окне            |
| `X-RateLimit-Reset`     | Unix-временная метка сброса окна               |
| `Retry-After`           | Секунд ожидания перед повтором (только на 429) |

## Исчерпание квоты

При превышении ежемесячной квоты плана:

```
HTTP/1.1 429 Too Many Requests

{
  "error": "quota_exceeded",
  "message": "Monthly search-unit quota exhausted. Upgrade your plan or wait for reset."
}
```

Поисковые единицы сбрасываются 1-го числа каждого календарного месяца. Обновите план в **Settings → Billing**,
чтобы немедленно восстановить квоту.

## Стратегия повторов для клиентов

| Класс ошибки              | Действие                                                                                 |
| ------------------------- | ---------------------------------------------------------------------------------------- |
| 4xx (кроме 429)           | **Не повторять** — исправьте запрос                                                      |
| 429 `rate_limit_exceeded` | Подождите секунды из `Retry-After`, затем повторите                                      |
| 429 `quota_exceeded`      | Не повторять до сброса квоты или обновления плана                                        |
| 502 `search_failed`       | Повторите один раз через 1 секунду; если снова не удалось — покажите ошибку пользователю |
| 503                       | Повторите с экспоненциальной задержкой: 1 с → 2 с → 4 с → отказ                          |
| Сетевая ошибка            | Повторите с экспоненциальной задержкой                                                   |

### Поведение SDK при повторах

SDK `@repo/search-client` не выполняет автоматические повторы при ошибках 5xx. Реализуйте логику повторов
в слое вашего приложения или используйте библиотеку вроде `p-retry`.

```typescript
import pRetry from "p-retry";

const results = await pRetry(() => client.search({ q: "headphones" }), {
	retries: 3,
	factor: 2,
	minTimeout: 1000,
	shouldRetry: (err) => err.status >= 500, // повторять только ошибки сервера
});
```

## Поведение модулей CMS при повторах

Буфер приёма данных AACsearch рассчитан на повторные попытки:

- Запросы синхронизации коннектора (`sync/full`, `sync/delta`) ставят документы в очередь `SearchIngestBuffer`
- Фоновый воркер повторяет неудавшиеся строки с экспоненциальной задержкой
- Модулю CMS не нужно реализовывать повтор для отдельных документов — буфер берёт это на себя
- Модуль CMS должен повторять сам HTTP-запрос при ответах 5xx (но не на 4xx)

## Отладка при разработке

При ошибках 502 во время разработки:

1. Проверьте, работает ли Typesense: `curl http://localhost:8108/health`
2. Проверьте логи API на наличие исходной ошибки Typesense (она сопоставляется перед отправкой ответа клиенту)
3. Проверьте `TYPESENSE_HOST`, `TYPESENSE_PORT`, `TYPESENSE_API_KEY` в `.env.local`

При ошибках 401:

1. Убедитесь, что срок действия ключа не истёк
2. Убедитесь, что префикс ключа соответствует операции (`ss_search_*` для поиска, `ss_connector_*` для приёма данных)
3. Для ограниченных токенов: проверьте поле `expiresAt` в декодированной нагрузке
