AACsearch раскрывает три различные API-поверхности. Понимание того, какую из них использовать в каком контексте, важно для безопасности и исключения вызовов неверного эндпоинта.

## Три API-поверхности

### 1. oRPC (процедуры панели управления / администрирования)

Используется панелью SaaS AACsearch. Требует сессии Better Auth (аутентифицированный пользователь).
Не предназначен для внешних вызывающих сторон или модулей CMS.

- **Базовый путь:** `/api/rpc/**`
- **Auth:** Сессионная кука (Better Auth)
- **Используется:** панелью `apps/saas`, операциями управления
- **26 процедур** — индексы, ключи, токены, аналитика, коннекторы, релевантность

### 2. Публичный поисковый эндпоинт

Используется браузерами, браузерными SDK и хостинговым виджетом. Требует ключа `ss_search_*` или `ss_scoped_*`.

- **Базовый путь:** `/api/search` (смонтирован в `public-handler.ts`)
- **Auth:** Bearer-токен (API-ключ в заголовке Authorization)
- **Используется:** `@repo/search-client`, `packages/widget`, любым HTTP-клиентом
- **Операции:** поиск, мульти-поиск

### 3. Connector API

Используется модулями CMS (PrestaShop, Bitrix, пользовательскими интеграциями). Требует ключа `ss_connector_*`.

- **Базовый путь:** `/api/connector/**`
- **Auth:** Bearer-токен (ключ `ss_connector_*`)
- **Используется:** модулями CMS, пользовательскими скриптами синхронизации
- **Операции:** handshake, heartbeat, sync.full, sync.delta, delete, diagnostics, sync status

## REST API v1

В дополнение к вышеперечисленному AACsearch раскрывает REST API v1 по адресу `/api/v1/` со спецификацией OpenAPI 3.1.

**Текущий статус:** Поставлен. 15 эндпоинтов — проекты, индексы, документы, поиск, управление API-ключами.

Спецификация OpenAPI: `GET /api/v1/openapi.json`

Области аутентификации для v1:

| Префикс       | Область  | Уровень доступа       |
| ------------- | -------- | --------------------- |
| `aa_admin_*`  | `admin`  | Полное управление     |
| `aa_write_*`  | `write`  | Приём данных + запись |
| `aa_search_*` | `search` | Только поиск          |
| `aa_scoped_*` | `scoped` | Ограниченный поиск    |

## Формат запроса (публичный поиск)

Все поисковые запросы — это JSON POST:

```
POST /api/search
Authorization: Bearer ss_search_your_key
Content-Type: application/json

{
  "indexSlug": "products",
  "q": "wireless headphones",
  ...параметры поиска...
}
```

## Поток аутентификации

```
Запрос поступает
    │
    ▼
Извлечь Authorization: Bearer {token}
    │
    ▼
Определить тип ключа по префиксу:
  ss_search_*    → проверить хэш API-ключа, проверить источник, проверить лимит запросов, проверить квоту
  ss_scoped_*    → проверить подпись HMAC, извлечь ограниченный фильтр, продолжить как поиск
  ss_connector_* → проверить хэш API-ключа, проверить область connector_write
    │
    ▼
Перейти к обработчику (поиск / приём данных / операция коннектора)
```

## Формат ошибок

Все ошибки из публичного поискового эндпоинта следуют этому формату:

```json
{
	"error": "error_code",
	"message": "Human-readable description"
}
```

Распространённые коды ошибок: `unauthorized`, `forbidden`, `rate_limit_exceeded`, `quota_exceeded`,
`index_not_found`, `search_failed`, `ingest_failed`.

Сырые ошибки Typesense **никогда** не пересылаются вызывающим сторонам. Все ошибки на стороне сервера сопоставляются с типизированными кодами перед отправкой ответа.
