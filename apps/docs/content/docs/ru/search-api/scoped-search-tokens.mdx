Ограниченные токены позволяют создавать краткосрочные, криптографически подписанные поисковые токены, сужающие то, что может видеть вызывающая сторона. Генерируйте их на стороне сервера и передавайте в браузер — базовый ключ `ss_search_*` никогда не покидает ваш сервер.

## Когда использовать ограниченные токены

| Сценарий                                                             | Использовать ограниченный токен?        |
| -------------------------------------------------------------------- | --------------------------------------- |
| Одноарендная витрина (одна организация, один магазин)                | Нет — ключа `ss_search_*` достаточно    |
| Мультиарендное приложение, где пользователи видят только свои товары | **Да**                                  |
| Предоставление поиска третьей стороне с ограниченным доступом        | **Да**                                  |
| Ограничение результатов конкретным ценовым сегментом                 | **Да**                                  |
| Краткосрочное встраивание виджета с истечением                       | **Да**                                  |
| Превью внутренней панели управления                                  | Нет — используйте oRPC на основе сессии |

## Как работают ограниченные токены

Ограниченные токены — это stateless HMAC-SHA256-подписанные утверждения — они никогда не хранятся в базе данных.

```
Структура токена:
  ss_scoped_{base64url(payload)}.{HMAC-SHA256 signature}

Нагрузка:
  {
    "organizationId": "org_...",
    "indexSlug": "products",
    "scopedFilter": "availability:=in_stock",
    "issuedAt": 1717200000,
    "expiresAt": 1717203600   // опционально
  }

Подпись:
  HMAC-SHA256(payload, BETTER_AUTH_SECRET)
```

Подпись верифицируется на стороне сервера при каждом запросе. Подделка нагрузки (например, удаление
`scopedFilter`) аннулирует подпись и даёт ответ 401.

## Генерация ограниченного токена

### Через oRPC (сторона сервера)

```typescript
// Server Component, Server Action или API route
const scopedToken = await orpc.search.createScopedToken.call({
	organizationId: session.organizationId,
	indexSlug: "products",
	scopedFilter: "price:<100", // всегда объединяется через AND с фильтрами вызывающей стороны
	expiresInSeconds: 3600, // TTL 1 час
	name: "Budget search for user XYZ", // опциональная метка для аудита
});

// scopedToken.token: "ss_scoped_abc...123" — передайте в браузер
```

### Через панель управления

1. Перейдите в **Search** → **API Keys** → **Scoped Tokens**
2. Нажмите **Create scoped token**
3. Введите выражение `scopedFilter`
4. Задайте срок истечения
5. Скопируйте сгенерированный токен

Примечание: токены, созданные через панель управления, предназначены для тестирования. Производственные токены всегда должны
генерироваться динамически для каждого пользователя в обработчике на стороне сервера.

## Использование ограниченного токена в браузере

Ограниченные токены используются точно так же, как обычные ключи поиска:

```typescript
import { AacSearchClient } from "@repo/search-client";

// Токен получен от вашего сервера (например, через Server Action или API route)
const client = new AacSearchClient({
	baseUrl: process.env.NEXT_PUBLIC_API_URL,
	apiKey: scopedToken,
	indexSlug: "products",
});

const results = await client.search({
	q: "headphones",
	filterBy: "brand:=Sony", // Фильтр вызывающей стороны: только бренд Sony
	// AACsearch объединит это через AND с "price:<100" из токена
	// Эффективный фильтр: "brand:=Sony && price:<100"
});
```

## Комбинирование фильтров

`scopedFilter` из токена всегда объединяется через AND с `filterBy` вызывающей стороны.
Вызывающая сторона не может удалить или обойти фильтр токена.

```
filterBy вызывающей стороны: "brand:=Sony"
scopedFilter токена:         "price:<100"
Эффективный фильтр:          "brand:=Sony && price:<100"
```

Это реализовано в `combineFilters()` в `packages/api/modules/search/lib/scoped-token.ts`.
Обход этой функции является **нарушением Hard Invariant** в кодовой базе.

## Истечение срока действия

Установите `expiresInSeconds`, чтобы ограничить срок действия токена. По истечении токен возвращает 401.

Рекомендуемые значения TTL:

| Контекст                  | TTL                           |
| ------------------------- | ----------------------------- |
| Сессия виджета            | 1–4 часа                      |
| Одна загрузка страницы    | 15–30 минут                   |
| Тестирование / разработка | Без истечения (опустите поле) |

Истёкшие токены отклоняются на стороне сервера — никакого таймера на стороне клиента или механизма обновления не нужно.

## Шаблон ротации

Для токенов, ограниченных сессией, генерируйте новый токен при каждом установлении пользовательской сессии:

```typescript
// Next.js middleware или server action, вызываемый при старте сессии
export async function generateSearchToken(session: Session) {
	return await orpc.search.createScopedToken.call({
		organizationId: session.organizationId,
		indexSlug: "products",
		scopedFilter: `organization_id:=${session.organizationId}`,
		expiresInSeconds: 4 * 60 * 60, // 4 часа
	});
}
```

Храните токен на клиенте (например, в React context, Zustand store) на протяжении сессии.

## Модель безопасности

- Ограниченные токены подписываются над `BETTER_AUTH_SECRET` — храните этот секрет в безопасности
- Нагрузка токена закодирована в base64, но не зашифрована — воспринимайте её как непрозрачную для браузера
- Не помещайте чувствительные данные в выражение `scopedFilter` (оно видно в нагрузке токена)
- Для очень чувствительных фильтров используйте поиск на стороне сервера вместо ограниченного токена

## Ограничения

- Ограниченные токены могут только **сужать** разрешения — они не могут предоставить доступ к индексам или организациям,
  к которым базовый ключ уже не имеет доступа
- Ограниченный токен привязан к одному слагу индекса
- Ограниченные токены не поддерживают операции `connector_write` — они предназначены только для поиска
