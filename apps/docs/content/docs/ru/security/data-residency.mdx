# Хранение данных (Data Residency)

AACsearch предлагает мультирегиональное хранение данных, чтобы вы могли выбрать, где будут храниться и обрабатываться ваши поисковые данные. Это критически важно для соблюдения региональных законов о защите данных, таких как GDPR, 152-ФЗ и требований SOC 2.

## Доступные регионы

| Регион | Код  | Расположение        | Соответствие нормам |
| ------ | ---- | ------------------- | ------------------- |
| Европа | `eu` | Франкфурт, Германия | GDPR                |
| США    | `us` | Вирджиния, США      | SOC 2, CCPA         |
| Россия | `ru` | Москва, Россия      | 152-ФЗ, 242-ФЗ      |

Каждый регион работает как полностью независимый стек AACsearch со своими:

- **Кластером Typesense** — все поисковые индексы и данные
- **База данных PostgreSQL** — метаданные организации, API-ключи и записи использования
- **Объектное хранилище** — вложения документов, файлы базы знаний и буферы загрузки

**Межрегиональный обмен данными отсутствует**. Данные, хранящиеся в одном регионе, никогда не покидают границ инфраструктуры этого региона.

## Регион по умолчанию

Новые организации автоматически назначаются в регион **EU (Франкфурт)**. Это обеспечивает соответствие GDPR для всех новых клиентов.

## Выбор региона

Вы можете выбрать регион хранения при настройке организации или изменить его позже на странице **Compliance** в панели управления.

**Рекомендации при выборе региона:**

- **Задержка**: Поисковые запросы направляются в кластер Typesense в выбранном регионе. Для лучшей производительности выбирайте регион, ближайший к вашим конечным пользователям.
- **Соответствие нормам**: Убедитесь, что регион соответствует требованиям вашей организации (GDPR, 152-ФЗ, SOC 2 и т.д.).
- **Существующие клиенты**: Если вам нужно перенести существующие данные в новый регион, используйте инструмент переноса данных (см. ниже).

## Изменение региона

Чтобы изменить регион хранения данных:

1. Перейдите в **Настройки организации → Compliance**
2. В разделе **Хранение данных (Data Residency)** просмотрите текущий регион
3. Выберите новый регион из выпадающего списка
4. Нажмите **Обновить регион** для сохранения

> **Важно**: Изменение региона обновляет маршрутизацию только для _новых_ данных. Существующие поисковые данные остаются в исходном регионе до запуска переноса.

## Перенос данных между регионами

При изменении региона хранения существующие поисковые данные остаются в предыдущем регионе. Для их переноса:

1. После изменения региона появится карточка **Перенести существующие данные**
2. Нажмите **Начать перенос** для копирования всех поисковых данных из старого региона в новый
3. Если выбрана опция **Обновить регион после переноса**, инструмент автоматически обновит настройки организации после завершения

Перенос использует подход **копирование и проверка**:

- Данные экспортируются из исходного кластера Typesense
- Каждый документ импортируется в целевой кластер
- Исходные данные сохраняются до подтверждения успешного переноса
- Прогресс отображается для каждого поискового индекса

**Что переносится:**

- Все поисковые документы во всех индексах
- Схемы и конфигурации поисковых индексов
- Наборы синонимов

**Что НЕ переносится:**

- API-ключи поиска (хранятся в базе данных, не в Typesense)
- Логи использования и данные аналитики
- Метаданные организации (название, участники и т.д.)

## Мониторинг работоспособности региона

Страница Compliance показывает статус вашего текущего региона, включая:

- Статус (онлайн/офлайн) кластера Typesense
- Текущую задержку (мс)
- Сообщения об ошибках, если кластер недоступен

## CDN и кэширование на границе сети по регионам

AACsearch поддерживает региональную конфигурацию CDN. Каждый регион может иметь свой слой кэширования:

### Рекомендуемая настройка

1. **DNS-конфигурация**: Настройте гео-маршрутизируемые DNS-записи (через AWS Route 53 или Cloudflare), направляющие пользователей к ближайшей CDN.
2. **CDN**: Разверните дистрибутив CDN (CloudFront, Cloudflare или Fastly) для каждого региона.
3. **API Gateway**: AACsearch использует региональную маршрутизацию — запрос направляется в правильный кластер Typesense на основе региона организации.
4. **Правила кэширования**: Настройте короткий TTL (30-60 сек) для результатов поиска и более длинный (5-15 мин) для метаданных коллекций.

## Справочник API

### Список доступных регионов

```http
GET /api/compliance/regions
```

### Получение региона организации

```http
GET /api/compliance/organizations/{organizationId}/region
```

### Установка региона организации

```http
PUT /api/compliance/organizations/{organizationId}/region
Content-Type: application/json

{
  "region": "us"
}
```

### Перенос данных организации

```http
POST /api/compliance/organizations/{organizationId}/migrate
Content-Type: application/json

{
  "sourceRegion": "eu",
  "destRegion": "us",
  "updateRegionAfterMigration": true
}
```

### Проверка работоспособности регионов

```http
GET /api/compliance/regions/health
```
