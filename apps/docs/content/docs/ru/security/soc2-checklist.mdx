# Контрольный список подготовки к SOC2 Type II

Этот список охватывает все пять критериев доверия SOC2 для платформы search-as-a-service AACsearch. Используйте его для отслеживания готовности перед привлечением внешнего аудитора для сертификации Type II.

<Callout type="info">
	**Целевой период аудита:** 6–12 месяцев сбора доказательств. Каждый пункт должен быть
	проверяемым с помощью системных журналов, снимков конфигурации или документов политик.
</Callout>

## Безопасность

Принцип безопасности защищает от несанкционированного доступа (как физического, так и логического).

### Контроль доступа

- [ ] **RBAC внедрен во всех системах** — PostgreSQL, Typesense, Coolify и S3 применяют управление доступом на основе ролей. Убедитесь, что каждая учетная запись человека и сервиса имеет минимально необходимые разрешения.
- [ ] **MFA обязательна для всех административных учетных записей** — Администратор Coolify, облачный провайдер (AWS/Vercel) и организация GitHub требуют аппаратный TOTP или WebAuthn.
- [ ] **Политика ротации ключей API документирована** — Ключи API AACsearch следуют графику ротации в 90 дней. Ключи ограничены по проектам с детализацией чтения/записи/удаления.
- [ ] **JIT-доступ к продакшену** — Нет постоянного административного доступа к производственным базам данных или поисковым кластерам. Доступ запрашивается через PagerDuty или Coolify с автоматическим истечением через 4 часа.
- [ ] **Ежеквартальная проверка доступа** — Все SSH-ключи, токены API и роли IAM проверяются каждый квартал. Отозванные ключи регистрируются в аудиторском журнале.

### Шифрование

- [ ] **TLS 1.3 обязателен для всех конечных точек API** — Все общедоступные конечные точки API AACsearch (поиск, индексация, управление) используют TLS 1.3. Автоматическое обновление сертификатов через Let's Encrypt или ACME проверяется.
- [ ] **Шифрование в состоянии покоя на всех томах хранения** — RDS (PostgreSQL) использует AES-256; корзины S3 имеют включенный SSE-S3; каталог данных Typesense находится на зашифрованном томе EBS.
- [ ] **Данные индекса клиента зашифрованы по арендаторам** — Подтвердите, что ключи шифрования на уровне индекса (если реализованы) изолированы между проектами.
- [ ] **Управление ключами документировано** — Мастер-ключи хранятся в AWS KMS (или эквиваленте). Проверяются политика ротации ключей, журналирование доступа и резервное копирование материала ключей.

### Управление уязвимостями

- [ ] **Автоматическое сканирование зависимостей** — Dependabot или Renovate настроены во всех репозиториях (supastarter, aacflow, infrastructure-as-code). Критические CVE исправляются в течение 72 часов.
- [ ] **Сканирование образов контейнеров** — Docker-образы для Typesense и любых пользовательских сервисов сканируются с помощью Trivy или AWS ECR перед развертыванием.
- [ ] **Ежегодное пентестирование** — Сторонний пентест, нацеленный на API AACsearch, инъекцию запросов Typesense и обход аутентификации. План устранения задокументирован для каждой находки.
- [ ] **Политика управления исправлениями** — Исправления уровня ОС применяются к вычислительным экземплярам в течение 30 дней после выпуска. Экстренные исправления (CVSS >= 9.0) применяются в течение 48 часов.

### Обнаружение вторжений

- [ ] **Правила WAF активны** — Межсетевой экран веб-приложений (Cloudflare или AWS WAF) блокирует SQL-инъекции, XSS и path traversal. Ограничение скорости настроено по ключу API и по IP.
- [ ] **Агрегация журналов и оповещения** — Все журналы приложений, баз данных и инфраструктуры отправляются в SIEM (например, Grafana Loki, Datadog или AWS CloudWatch). Настроены оповещения для: повторных неудачных входов, необычного объема вызовов API и известных вредоносных IP-паттернов.
- [ ] **Мониторинг целостности файлов** — Критические каталоги двоичных файлов и конфигураций (каталог данных Typesense, секреты приложения, конфигурация nginx) отслеживаются на предмет несанкционированных изменений.
- [ ] **Cloud Trail включен** — AWS CloudTrail или эквивалент включен на всех учетных записях с журналированием событий чтения/записи. Журналы неизменны в течение как минимум 12 месяцев.

## Доступность

Принцип доступности гарантирует, что система работоспособна и доступна в соответствии с обязательствами по SLA.

### Мониторинг времени безотказной работы

- [ ] **Синтетический мониторинг настроен** — Внешние зонды (Checkly, Better Uptime или эквивалент) проверяют конечные точки поиска и индексации каждые 60 секунд из нескольких географических регионов.
- [ ] **Страница статуса настроена** — Публичная страница статуса (например, status.aacsearch.com) отражает состояние компонентов в реальном времени с временной шкалой инцидентов.
- [ ] **Обязательства по SLA задокументированы** — Опубликованный SLA по времени безотказной работы (99.9% или выше) с определенным окном измерения, окнами исключений и условиями кредитования.
- [ ] **Мультирегиональное аварийное переключение протестировано** — Если существует мультирегиональное развертывание, переключение между регионами тестируется ежеквартально. RTO для регионального переключения задокументирован (< 15 минут цель).

### План аварийного восстановления

- [ ] **План аварийного восстановления задокументирован и протестирован** — Письменный план DR, охватывающий полное восстановление PostgreSQL (PITR через WAL-G), снимков Typesense и состояния приложения. Командные учения проводятся каждые 6 месяцев.
- [ ] **RTO и RPO определены** — Целевое время восстановления: 1 час. Целевая точка восстановления: 15 минут. Оба значения проверяемы с помощью учебных восстановлений.
- [ ] **Межрегиональное восстановление протестировано** — Как минимум одно полное учебное восстановление выполняется ежегодно во вторичном регионе с использованием сохраненных резервных копий.
- [ ] **План коммуникации задокументирован** — Список уведомления заинтересованных сторон, дерево эскалации и шаблон посмертного анализа инцидентов включены в план DR.

### Проверка резервных копий

- [ ] **Автоматические резервные копии активны** — PostgreSQL: непрерывное архивирование WAL-G в S3. Typesense: API снимков каждые 6 часов в S3. Приложение: на основе Git (GitHub) + Coolify.
- [ ] **Целостность резервных копий проверяется ежемесячно** — Автоматическое тестовое восстановление последнего снимка Typesense и резервной копии PostgreSQL в среде staging. Результаты регистрируются и проверяются.
- [ ] **Политика хранения резервных копий задокументирована** — Почасовые снимки хранятся 7 дней, ежедневные — 30 дней, еженедельные — 12 месяцев. Межрегиональная копия еженедельных резервных копий.
- [ ] **Шифрование резервных копий проверено** — Все данные резервных копий в S3 зашифрованы (SSE-S3 или SSE-KMS). Процесс восстановления проверяет успешность расшифровки без ручного вмешательства с ключами.

### Реагирование на инциденты

- [ ] **План реагирования на инциденты задокументирован** — Письменный план IR с уровнями серьезности (SEV-1 — SEV-4), временем реагирования и путями эскалации. Пересматривается ежегодно.
- [ ] **График дежурств установлен** — Ротация PagerDuty или Opsgenie покрывает 24/7. Эскалация через 15 минут после отсутствия подтверждения.
- [ ] **Посмертный анализ инцидентов проводится** — Каждый инцидент SEV-1 и SEV-2 имеет письменный посмертный анализ с первопричиной, действиями и оценкой влияния на SLA.
- [ ] **Runbooks автоматизированы** — Распространенные типы инцидентов (отказ БД, деградация поискового кластера, замедление API) имеют автоматизированные runbooks, запускающие корректирующие действия.

## Целостность обработки

Целостность обработки гарантирует, что обработка системы является полной, корректной, точной, своевременной и авторизованной.

### Проверка входных данных

- [ ] **Валидация запросов API** — Все поисковые запросы, полезные нагрузки индексации и запросы к API управления проверяются на соответствие JSON-схеме. Недействительные запросы отклоняются с описательными кодами ошибок до достижения Typesense.
- [ ] **Ограничения размера индекса применяются** — Ограничения количества документов и размера для каждого проекта применяются на шлюзе API. Чрезмерные полезные нагрузки отклоняются со статусом 413.
- [ ] **Очистка поисковых запросов** — Запросы Typesense очищаются для предотвращения инъекций. Специальные символы в текстовых полях экранируются или удаляются в соответствии с задокументированными правилами.
- [ ] **Ограничение скорости на конечную точку** — Различные ограничения скорости применяются к конечным точкам поиска (выше) и индексации (ниже). Лимиты задокументированы в публичной API-справке и применяются через промежуточное ПО.

### Конвейеры обработки данных

- [ ] **Конвейер индексации документов отслеживается** — Сквозная задержка индексации документов (получение API → подтверждение индекса) отслеживается. Настроены оповещения, если P99 превышает 5 секунд.
- [ ] **Поддерживаются ключи идемпотентности** — API индексации поддерживает ключи идемпотентности, чтобы повторные попытки не создавали дубликаты документов. Реализация проверена через интеграционные тесты.
- [ ] **Целостность пакетной обработки** — Пакетные операции с документами (создание, обновление, удаление) являются атомарными для каждого пакета. Частичные сбои откатывают весь пакет. Подтверждено через журналы аудита.
- [ ] **Журналы преобразования данных** — Любое обогащение или преобразование, применяемое к документам перед индексацией, регистрируется. Правила преобразования имеют версионность в Git.

### Обработка ошибок

- [ ] **Плавная деградация** — API поиска возвращает устаревшие или деградированные результаты (с флагом `degraded: true`) вместо ошибок 500, когда нижестоящие зависимости (например, кэш, вторичный индекс) недоступны.
- [ ] **Структурированные ответы об ошибках** — Все ошибки API следуют согласованной структуре JSON: `{ error: { code, message, requestId } }`. Внутренние трассировки стека никогда не раскрываются клиентам.
- [ ] **Очередь недоставленных сообщений настроена** — Неудачные задания индексации, доставки вебхуков и асинхронные задачи обработки направляются в DLQ. DLQ отслеживается и оповещается.
- [ ] **Бюджет ошибок определен** — Бюджет ошибок (например, 0.1% запросов могут завершаться ошибкой) определен для каждого SLA. Оповещения срабатывают при 50% потреблении бюджета ошибок и повторно при 90%.

## Конфиденциальность

Конфиденциальность защищает конфиденциальную информацию в соответствии с соглашениями.

### Классификация данных

- [ ] **Политика классификации данных задокументирована** — Определены категории: Публичные, Внутренние, Конфиденциальные, Ограниченные. Поисковые индексы клиентов по умолчанию классифицируются как Конфиденциальные.
- [ ] **Маркировка данных применяется** — Теги корзин S3, комментарии столбцов базы данных и заголовки ответов API включают метки классификации, где это применимо.
- [ ] **Процедуры обработки данных** — Письменные процедуры для каждого уровня классификации, охватывающие хранение, передачу, хранение и уничтожение. Пересматриваются ежегодно.

### Журналирование доступа

- [ ] **Весь доступ к данным клиентов регистрируется** — Каждый вызов API, который читает или записывает данные поискового индекса, регистрируется с: временной меткой, субъектом (ключ API или ID пользователя), действием, ресурсом, исходным IP и статусом ответа.
- [ ] **Хранение журналов соответствует требованиям аудита** — Журналы доступа хранятся не менее 12 месяцев (или согласно договорным обязательствам). Журналы доступны только для добавления и неизменны.
- [ ] **Обнаружение аномалий в паттернах доступа** — Автоматические оповещения о необычных паттернах доступа: одни учетные данные обращаются ко многим индексам, массовый экспорт в нерабочее время или повторяющиеся ответы 403.
- [ ] **Мониторинг привилегированного доступа** — Весь административный доступ к производственной инфраструктуре (SSH, консоль базы данных, администратор Coolify) регистрируется отдельно с записью сессии, где это возможно.

### NDA для подрядчиков

- [ ] **NDA подписаны всеми подрядчиками** — Подписанные NDA хранятся для каждого подрядчика, поставщика и субподрядчика, имеющего доступ к системам AACsearch или данным клиентов.
- [ ] **Оценки рисков поставщиков завершены** — Критические поставщики (облачный провайдер, CDN, инструменты мониторинга) заполнили анкеты безопасности. Результаты проверены и приняты командой безопасности.
- [ ] **Договорные положения о защите данных** — Все контракты с поставщиками включают условия обработки данных, SLA уведомления о нарушениях и пункты о праве на аудит.

## Конфиденциальность личных данных

Конфиденциальность касается сбора, использования, хранения и раскрытия личной информации.

### Уведомление о конфиденциальности

- [ ] **Публичное уведомление о конфиденциальности опубликовано** — Политика конфиденциальности на aacsearch.com/privacy охватывает: какие PII собираются, как они используются, обмен данными, файлы cookie, права пользователей (доступ, удаление, переносимость) и контактную информацию.
- [ ] **Уведомление соответствует применимым нормативным требованиям** — Уведомление о конфиденциальности соответствует GDPR, CCPA и любым требованиям конкретной юрисдикции для клиентской базы AACsearch.
- [ ] **Версионность уведомления поддерживается** — Изменения в уведомлении о конфиденциальности имеют версионность. Существенные изменения сообщаются клиентам по электронной почте или в уведомлении в приложении в течение 30 дней.

### Управление согласием

- [ ] **Согласие собирается при регистрации** — Процесс регистрации пользователя включает явные флажки согласия на: принятие политики конфиденциальности, маркетинговые коммуникации (opt-in) и обработку данных для предоставления услуг.
- [ ] **Записи согласия хранятся** — Временные метки согласия, IP-адреса и версии сохраняются и доступны для запросов. Пользователи могут отозвать согласие и просмотреть свой текущий статус согласия в настройках учетной записи.
- [ ] **Баннер согласия на файлы cookie активен** — Баннер согласия на файлы cookie на всех публичных страницах (документация, маркетинговый сайт) с детальным контролем для аналитических и обязательных cookie.

### Хранение данных

- [ ] **График хранения данных задокументирован** — Четкий график для каждой категории данных: аналитика поиска (13 месяцев), журналы API (12 месяцев), учетные записи пользователей (до запроса на удаление), платежные записи (7 лет).
- [ ] **Автоматическая очистка данных** — Запланированные задания (cron или lambda) применяют ограничения хранения. Очистка регистрируется и проверяема.
- [ ] **Процесс удаления данных задокументирован** — По запросу клиента: поисковые индексы удаляются в течение 30 дней, данные учетной записи — в течение 60 дней. Процесс включает этап проверки и подтверждение клиенту.

### Обработка PII

- [ ] **Инвентаризация PII поддерживается** — Реестр всех полей данных PII на платформе: электронные письма пользователей, IP-адреса в журналах, платежные данные и любые PII, предоставленные клиентом в поисковых индексах.
- [ ] **Минимизация PII применяется** — Дизайн API ограничивает сбор PII строго необходимым. Клиентам рекомендуется не индексировать PII в поисковых документах (задокументировано в руководстве по лучшим практикам).
- [ ] **План уведомления о нарушениях** — Письменный план, охватывающий: триггеры обнаружения, внутреннюю эскалацию, юридическую проверку, уведомление клиента (в течение 72 часов в соответствии с GDPR) и уведомление регулирующих органов.
- [ ] **Оценка воздействия на защиту данных (DPIA)** — DPIA завершена для операций обработки с высоким риском. Пересматривается ежегодно или при значительных изменениях обработки.

## Контрольный список доказательств для аудитора

Подготовьте следующие артефакты перед аудитом Type II:

- [ ] Описание системы (описание архитектуры AACsearch, границ и средств контроля)
- [ ] Матрица контроля, сопоставляющая каждый контроль с критерием доверия
- [ ] Пакеты доказательств: 6–12 месяцев журналов, снимков конфигурации, документов политик
- [ ] Организационная диаграмма, показывающая роли и обязанности в области безопасности
- [ ] Отчет стороннего пентеста (не старше 12 месяцев)
- [ ] Отчеты об оценке рисков поставщиков для критических организаций субсервисов
- [ ] Завершенная оценка готовности к SOC2 Type II (самооценка или с помощью консультанта)
