Каждой SaaS-платформе рано или поздно требуется поиск. Вопрос в том, как его архитектурно реализовать: как встроенную функцию? Как API, который вы предоставляете клиентам? Или как управляемый сервис, который вы потребляете?

В этой статье рассматриваются архитектурные паттерны, которые работают в масштабе — независимо от того, встраиваете ли вы поиск в свой продукт или предлагаете его как функцию своим клиентам.

## Варианты архитектуры

### Вариант 1: Встроенный поиск (своими силами)

Вы реализуете поиск непосредственно в своей базе данных с помощью PostgreSQL `ILIKE` или Elasticsearch.

```
Ваше приложение → PostgreSQL `ILIKE` / Elasticsearch → Результаты
```

| Аспект                      | Оценка                                              |
| --------------------------- | --------------------------------------------------- |
| Время настройки             | 2-4 недели                                          |
| Обслуживание                | Постоянное (индексация, настройка, масштабирование) |
| Мультиарендность            | Нужно реализовать самостоятельно                    |
| Аналитика                   | Нужно реализовать самостоятельно                    |
| Стоимость при 1M документов | $200-500/мес сервер + 20ч/мес эксплуатация          |

Этот вариант подходит для продуктов на ранних стадиях с объёмом менее 10K документов. После этого порога эксплуатационные расходы превышают выгоду от простоты.

### Вариант 2: Встроенный + сторонний движок

Вы интегрируете поиск-как-сервис (например, AACsearch) в своё приложение через API. Провайдер занимается индексацией, поиском и инфраструктурой.

```
Ваше приложение → AACsearch API → Результаты
```

| Аспект                      | Оценка                                |
| --------------------------- | ------------------------------------- |
| Время настройки             | 1-2 дня                               |
| Обслуживание                | Минимальное (управляется провайдером) |
| Мультиарендность            | Встроенная (ограниченные токены)      |
| Аналитика                   | Встроенная                            |
| Стоимость при 1M документов | $99-249/мес                           |

Это оптимальный вариант для большинства SaaS-компаний. Вы получаете production-ready поиск без необходимости в выделенном инженере по поиску.

### Вариант 3: Предоставление поиска как API-продукта

Вы создаёте слой поискового API, который потребляют ваши клиенты. AACsearch может выступать в роли бэкенда.

```
Приложение клиента → Ваш поисковый API → AACsearch → Результаты
                                               ↑
                                   Ваш API управляет ограниченными токенами,
                                   лимитированием запросов и биллингом
```

| Аспект           | Оценка                                |
| ---------------- | ------------------------------------- |
| Время настройки  | 1-2 недели                            |
| Обслуживание     | Управление поверхностью API           |
| Мультиарендность | Ограниченные токены на уровне клиента |
| Аналитика        | Аналитика по каждому клиенту          |
| Доход            | Тарификация за поиск или за индекс    |

Это модель для платформ, которые предлагают поиск как функцию своим конечным пользователям (например, e-commerce платформы, CMS-провайдеры).

## Шаблоны проектирования REST API

### Внутренний поисковый API (Вариант 2)

```typescript
// Внутренняя поисковая конечная точка для вашего SaaS-приложения
app.post("/api/search", async (req, res) => {
	const { tenantId } = await authenticate(req);
	const { query, filters, page = 1 } = req.body;

	const scopedKey = await getTenantKey(tenantId);
	const results = await aacsearch.search({
		apiKey: scopedKey,
		index: "products",
		q: query,
		filter_by: filters,
		page,
	});

	res.json(results);
});
```

### Клиентский поисковый API (Вариант 3)

```typescript
// Поисковый API для конечных пользователей ваших клиентов
app.post("/api/v1/search", async (req, res) => {
	// Лимитирование запросов по ключу API клиента
	await rateLimiter.check(req.apiKey);

	// Ключ клиента отображается на его ограниченный токен AACsearch
	const scopedKey = await getScopedKey(req.apiKey);
	const results = await aacsearch.search({
		apiKey: scopedKey,
		index: req.body.index,
		q: req.body.q,
		filter_by: req.body.filter_by,
	});

	// Добавление учёта использования для биллинга
	await trackUsage(req.apiKey, results);

	res.json(results);
});
```

## Архитектура индексации

### Индексация в реальном времени (на основе вебхуков)

Для SaaS-приложений с часто меняющимися данными используйте вебхук-индексацию:

```typescript
// При создании/обновлении записи в вашем приложении
app.post("/webhooks/aacsearch/sync", async (req, res) => {
	const { event, data } = req.body;

	switch (event) {
		case "product.created":
		case "product.updated":
			await aacsearch.upsertDocument("products", data);
			break;
		case "product.deleted":
			await aacsearch.deleteDocument("products", data.id);
			break;
	}

	res.status(200).end();
});
```

### Пакетная индексация (для первичного импорта или ночной синхронизации)

Для больших наборов данных используйте пакетный импорт:

```typescript
async function batchImport(tenantId: string, documents: any[]) {
	// Разбивка на чанки по 10K для оптимальной производительности
	const chunks = chunk(documents, 10000);

	for (const chunk of chunks) {
		await aacsearch.importDocuments(
			"products",
			chunk.map((doc) => ({
				...doc,
				tenant_id: tenantId,
			})),
		);
	}
}
```

### Гибридный подход

Используйте вебхуки для обновлений в реальном времени и ночную пакетную синхронизацию для целостности:

```
В реальном времени: Вебхук → Добавление/обновление отдельных документов (< 100 мс)
Ночью:            Cron-задача → Полная переиндексация для обнаружения пропущенных изменений
Восстановление:   Ручной запуск → Перестроение индекса из дампа базы данных
```

## Стратегии фильтрации

### Фильтрация по арендатору

Всегда применяйте изоляцию арендаторов на уровне поискового движка, а не на уровне приложения:

```typescript
// ХОРОШО: Ограниченный API-токен принудительно применяет фильтр арендатора
const key = await aacsearch.createScopedKey({
	filter: `tenant_id:=:${tenantId}`,
});

// ПЛОХО: Фильтрация на уровне приложения (риск утечки)
const results = await aacsearch.search({
	filter_by: `tenant_id:${tenantId}`, // Если баг уберёт это — межарендаторская утечка
});
```

### Фильтрация по ролям

Для приложений с ролевым доступом (администратор vs. просматривающий):

```typescript
const filters: string[] = [];

if (user.role === "viewer") {
	filters.push("visibility:public");
} else if (user.role === "editor") {
	filters.push("visibility:public,internal");
}
// Администратор видит всё — без фильтра

const scopedKey = await aacsearch.createScopedKey({
	filter: filters.join(" && "),
});
```

## Оптимизация производительности

| Стратегия                 | Влияние                                 | Реализация                                              |
| ------------------------- | --------------------------------------- | ------------------------------------------------------- |
| Денормализация документов | Ускорение запросов в 2-5 раз            | Объединение связанных данных в один документ            |
| Ограничение полей         | Ускорение индексации на 30-50%          | Индексировать только поля, по которым ищут пользователи |
| Пагинация                 | Постоянное время ответа                 | Использовать курсорную, а не смещённую пагинацию        |
| Кэширование               | Снижение запросов на 50-80%             | Кэшировать топ-100 запросов на 60 секунд                |
| Ограничение фасетов       | Ускорение вычисления фасетов в 2-3 раза | Ограничить до 10 значений фасета на поле                |

## Мониторинг и оповещения

Ваш поисковый API должен быть так же наблюдаем, как и ваше приложение:

```typescript
// Мониторинг ключевых метрик
interface SearchMetrics {
	p50_latency_ms: number;
	p95_latency_ms: number;
	p99_latency_ms: number;
	queries_per_second: number;
	error_rate: number;
	zero_result_rate: number;
	cache_hit_rate: number;
}

// Настройка оповещений
const alerts = {
	p95_latency_ms: { warn: 150, critical: 300 },
	error_rate: { warn: 0.01, critical: 0.05 }, // 1% и 5%
	zero_result_rate: { warn: 0.1, critical: 0.25 }, // 10% и 25%
};
```

## Когда предлагать поиск как функцию

Если ваша платформа имеет несколько клиентов, каждому из которых нужен поиск, рассмотрите возможность предоставления его как функции продукта:

- **E-commerce платформа:** Каждому магазину нужен поиск по товарам
- **CMS:** Каждому сайту нужен поиск по контенту
- **Маркетплейс:** Каждому списку продавца нужна фильтрация
- **SaaS:** Каждой организации нужен внутриприложенческий поиск

В каждом случае ограниченные токены AACsearch превращают мультиарендный поиск в конфигурацию, а не в инженерный проект.
