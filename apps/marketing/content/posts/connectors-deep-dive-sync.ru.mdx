Самая сложная часть поиска — это не сам поиск, а поддержание индекса в синхронизации с источником истинных данных. Когда изменяется цена товара, обновляется статья в блоге или снимается с производства SKU, поисковый индекс должен отражать это изменение за секунды, а не за часы.

В этой статье рассматривается, как коннекторы AACsearch решают проблему синхронизации для различных источников данных.

## Проблема синхронизации

Без автоматической синхронизации команды прибегают к:

- **Ручному экспорту CSV:** Загрузка раз в день или неделю — данные всегда устаревшие
- **Переиндексации по cron:** Запуск каждые N часов — расточительно при редких изменениях
- **Полному переимпорту:** Дорого в масштабе, вызывает временную деградацию поиска

Коннекторы AACsearch решают эту проблему тремя подходами, в зависимости от вашего источника данных.

## Подход 1: Вебхук API (лучше всего для кастомных приложений)

Для кастомных приложений синхронизация через вебхуки является наиболее гибкой. Ваше приложение отправляет обновления документов в AACsearch при каждом изменении данных:

```typescript
// В вашем CRUD-обработчике
async function updateProduct(id: string, data: ProductData) {
	// 1. Обновление вашей базы данных
	await db.products.update({ where: { id }, data });

	// 2. Синхронизация с AACsearch в реальном времени
	await aacsearch.upsertDocument("products", {
		id,
		name: data.name,
		price: data.price,
		description: data.description,
		category: data.category,
		in_stock: data.stock > 0,
		updated_at: new Date().toISOString(),
	});
}
```

### Пакетные операции

Для массовых изменений или первичного импорта:

```bash
# Импорт 10 000 документов (2 сек)
curl -X POST "https://api.aacsearch.com/indices/products/documents/import" \
  -H "X-API-KEY: ss_admin_key" \
  -H "Content-Type: application/json" \
  -d @products.json

# Удаление по запросу
curl -X POST "https://api.aacsearch.com/indices/products/documents/delete" \
  -H "X-API-KEY: ss_admin_key" \
  -H "Content-Type: application/json" \
  -d '{"filter_by": "category:=:discontinued"}'
```

### Upsert vs. Полная переиндексация

| Операция              | Скорость    | Когда использовать                               |
| --------------------- | ----------- | ------------------------------------------------ |
| `upsert`              | < 100 мс    | Обновление одного документа в реальном времени   |
| `import`              | 5K док/сек  | Пакетный импорт, первичная загрузка              |
| `delete`              | < 50 мс     | Удаление одного документа                        |
| Полная переиндексация | Различается | Изменение схемы, миграция данных, восстановление |

## Подход 2: Готовые CMS-коннекторы

AACsearch поставляется с коннекторами для популярных e-commerce и CMS-платформ. Они автоматически обрабатывают сопоставление схем, инкрементальную синхронизацию и распространение удалений.

### Коннектор PrestaShop

Устанавливается через Composer. Коннектор PrestaShop встраивается в жизненный цикл событий PrestaShop:

```bash
composer require aacsearch/prestashop-connector
php bin/console aacsearch:sync --store=my-store.com
```

**Что синхронизируется автоматически:**

| Сущность           | Тип синхр.         | Индексируемые поля                                                         |
| ------------------ | ------------------ | -------------------------------------------------------------------------- |
| Товары             | В реальном времени | название, цена, описание, категория, изображение, атрибуты, статус наличия |
| Категории          | При изменении      | название, описание, родитель, изображение                                  |
| Производители      | При изменении      | название, описание, логотип                                                |
| Комбинации товаров | В реальном времени | атрибуты вариантов, цена, наличие в stock                                  |
| Клиенты            | Опционально        | имя, email, группа                                                         |
| Заказы             | Опционально        | статус, сумма, товары, дата                                                |

Коннектор использует `hookActionObjectAddAfter`, `hookActionObjectUpdateAfter` и `hookActionObjectDeleteAfter` PrestaShop для синхронизации изменений в реальном времени.

### Коннектор Bitrix24

Устанавливается через Bitrix Marketplace. Коннектор Bitrix синхронизирует:

```bash
# Установка через Bitrix CLI
php bitrix.php module:install aacsearch.connector
php bitrix.php aacsearch:configure --api-key=ss_admin_key
```

**Синхронизируемые сущности:** Товары, разделы, свойства, цены и остатки.
**Режим синхронизации:** В реальном времени через API событий Bitrix с 30-секундной задержкой для массовых операций.

### Коннектор WooCommerce

```bash
wp plugin install aacsearch-woocommerce --activate
wp aacsearch:sync --full
```

**Синхронизируемые сущности:** Товары, вариации, категории, теги, атрибуты, отзывы.
**Режим синхронизации:** В реальном времени через вебхуки WooCommerce (`woocommerce_product_updated` и др.).

## Подход 3: Отслеживание изменений базы данных (CDC)

Для приложений без поддержки вебхуков CDC захватывает изменения непосредственно из вашей базы данных:

```sql
-- AACsearch CDC читает из WAL или binlog вашей базы данных
-- Каждый INSERT, UPDATE и DELETE захватывается и синхронизируется

-- Пример: публикация PostgreSQL для CDC
CREATE PUBLICATION aacsearch_pub FOR TABLE products, categories, brands;
```

**Поддерживаемые базы данных:**

- PostgreSQL (через логическую репликацию / WAL)
- MySQL (через binlog)
- MongoDB (через oplog)

Коннектор CDC работает как sidecar-процесс, который отслеживает поток изменений вашей базы данных и отправляет обновления в AACsearch:

```
Ваша БД → CDC-коннектор → AACsearch API
   │                │
   │                └── Пакетная обработка изменений каждые 500 мс
   │
   └── Запись продолжается в обычном режиме — без влияния на производительность
```

**Задержка:** Изменения отражаются в поиске в течение 500 мс – 2 сек, в зависимости от конфигурации пакетной обработки.

## Сравнение архитектуры коннекторов

| Подход              | Время настройки | Реальное время | Точность   | Обслуживание | Лучше всего для                       |
| ------------------- | --------------- | -------------- | ---------- | ------------ | ------------------------------------- |
| Вебхук API          | 2-4 часа        | Да             | 100%       | Низкое       | Кастомные приложения, API             |
| Плагин PrestaShop   | 10 минут        | Да             | 100%       | Очень низкое | Магазины PrestaShop                   |
| Коннектор Bitrix    | 10 минут        | Да             | 100%       | Очень низкое | Каталоги Bitrix24                     |
| Плагин WooCommerce  | 10 минут        | Да             | 100%       | Очень низкое | Магазины WooCommerce                  |
| CDC                 | 1-2 дня         | Почти реальное | Высокая    | Среднее      | Устаревшие приложения, нет вебхук API |
| Cron-переиндексация | 1 час           | Нет            | Устаревшие | Низкое       | Прототипы, допустима низкая точность  |

## Разрешение конфликтов

Когда обновление документа поступает в то время, как предыдущее обновление всё ещё индексируется, AACsearch использует стратегию «последняя запись побеждает»:

```json
{
	"upsert_strategy": "last_write_wins",
	"conflict_resolution": "timestamp_based"
}
```

Для приложений, требующих транзакционной согласованности, используйте вебхук API с ключами идемпотентности:

```typescript
await aacsearch.upsertDocument("products", data, {
	idempotencyKey: `product_${productId}_${version}`,
});
```

## Мониторинг вашего конвейера синхронизации

Здоровый конвейер синхронизации критически важен. Отслеживайте:

| Метрика                         | Здоровый диапазон            | Действие при выходе за пределы                        |
| ------------------------------- | ---------------------------- | ----------------------------------------------------- |
| Документов проиндексировано/сек | > 50                         | Проверьте конфигурацию коннектора, уменьшите пакет    |
| Задержка синхронизации          | < 2 сек                      | Проверьте сеть, увеличьте окно пакетной обработки CDC |
| Доля ошибок                     | < 0,1%                       | Проверьте несоответствия схем документов              |
| Глубина очереди                 | < 10K                        | Увеличьте размер пакета, масштабируйте коннектор      |
| Свежесть индекса                | < 5 мин с момента обновления | Проверьте работоспособность cron/вебхуков             |
