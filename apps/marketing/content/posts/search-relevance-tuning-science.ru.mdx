Релевантность поиска — это разница между «почти подошло» и «именно то, что нужно». Google тратит на это миллиарды. Ваши пользователи ожидают того же от вашего сайта.

Этот пост объясняет науку, стоящую за релевантностью поиска в AACsearch (работающем на Typesense) — от базового TF-IDF ранжирования до продвинутых методов настройки, которыми команды мерчандайзинга могут управлять без участия разработчиков.

## Основа: TF-IDF скоринг

Каждому поисковому движку нужен способ оценки документов. Typesense использует TF-IDF (Term Frequency-Inverse Document Frequency — частота термина, обратная частота документа) как базовую формулу ранжирования.

**Частота термина (Term Frequency):** Сколько раз поисковый термин встречается в документе? Товар с названием «Wireless Bluetooth Headphones», который упоминает «Bluetooth» трижды, ранжируется выше для запроса «Bluetooth», чем товар, упоминающий его один раз.

**Обратная частота документа (Inverse Document Frequency):** Насколько редок термин среди всех документов? «Bluetooth» более специфичен, чем «product» — совпадение по «Bluetooth» даёт больший вклад в релевантность, чем совпадение по «product», который встречается почти в каждом документе.

Typesense объединяет их в оценку `_text_match`, которая является полем сортировки по умолчанию. Более высокий `_text_match` = более релевантно.

```json
// Каждый результат поиска включает оценку релевантности
{
	"hits": [
		{
			"document": { "name": "Wireless Bluetooth Headphones" },
			"text_match": 142857,
			"text_match_info": {
				"best_field_score": "85678",
				"best_field_weight": 3,
				"fields_matched": 2
			}
		}
	]
}
```

## Веса полей: приоритизация значимых совпадений

Не все поля равны. Совпадение в названии товара `name` важнее, чем совпадение в `description`. AACsearch позволяет задавать веса для каждого поля:

| Поле        | Вес  | Обоснование                                                          |
| ----------- | ---- | -------------------------------------------------------------------- |
| name        | 3x   | Основной идентификатор — точное совпадение здесь сильнейший сигнал   |
| brand       | 2x   | Совпадение по бренду указывает на высокую заинтересованность         |
| category    | 2x   | Совпадение по категории подтверждает контекст просмотра пользователя |
| description | 1x   | Вспомогательное совпадение, но не такое сильное                      |
| tags        | 1.5x | Курируемые метаданные выявляют релевантные ассоциации                |
| sku         | 5x   | Точный поиск по SKU должен ранжироваться #1                          |

Настройте веса в Dashboard AACsearch в разделе Relevance Tuning:

```json
{
	"searchable_fields": [
		{ "name": "name", "boost": 3 },
		{ "name": "brand", "boost": 2 },
		{ "name": "category", "boost": 2 },
		{ "name": "description", "boost": 1 },
		{ "name": "tags", "boost": 1.5 }
	]
}
```

## Бонус за точное совпадение

Когда запрос точно совпадает со значением поля, Typesense даёт бонус к релевантности. Это критически важно для:

- **Поиска по брендам:** «Nike» точно совпадает с полем brand → товары Nike ранжируются #1
- **Названий товаров:** «iPhone 16» точно совпадает → конкретный товар показывается первым
- **SKU/Модельных номеров:** «[B07YZN5LLK]» точно совпадает → один результат, всегда первый

Бонус за точное совпадение включён по умолчанию. Настройте множитель в настройках Dashboard (по умолчанию: 2x).

## Настройка толерантности к опечаткам

Толерантность к опечаткам — не бинарная опция, а скользящая шкала. AACsearch позволяет управлять:

```json
{
	"typo_tolerance": {
		"enabled": true,
		"min_length": 4, // Запросы < 4 символов: без исправления опечаток
		"max_typos_per_word": {
			"1": 1, // Слова из 1–3 символов: 0 опечаток
			"4": 1, // Слова из 4–6 символов: макс. 1 опечатка
			"7": 2 // Слова из 7+ символов: макс. 2 опечатки
		}
	}
}
```

Это позволяет избежать избыточных исправлений. Поиск «iphon» (1 опечатка от «iphone») должен исправиться на iPhone. Но «socks» не должен возвращать «soccer» — это 2 опечатки, и контекст неверный.

## Группы синонимов: человеческий фактор

Алгоритмическая релевантность даёт 80% результата. Синонимы дают оставшиеся 20% — и именно здесь важна экспертиза в предметной области.

### Как работают синонимы в AACsearch

Группа синонимов — это набор эквивалентных терминов. Поиск любого термина из группы возвращает результаты, совпадающие с любым другим термином в группе:

```json
{
	"synonyms": [
		["sneakers", "trainers", "athletic shoes", "running shoes"],
		["laptop", "notebook", "macbook", "ultrabook"],
		["sofa", "couch", "settee", "loveseat"],
		["cell phone", "mobile phone", "smartphone", "iPhone"],
		["soda", "pop", "cola", "soft drink"],
		["hoodie", "hooded sweatshirt", "pullover"]
	]
}
```

### Типы синонимических связей

| Тип              | Пример                     | Эффект                                             |
| ---------------- | -------------------------- | -------------------------------------------------- |
| Однонаправленные | «laptop» → «notebook»      | Поиск «laptop» возвращает notebook; не наоборот    |
| Эквивалентные    | «sneakers» ↔ «trainers»    | Оба термина возвращают одни и те же результаты     |
| Иерархические    | «fruit» → [список фруктов] | Общий термин расширяется до конкретных позиций     |
| Брендовые        | «Nike» → «Air Max»         | Запрос бренда выводит конкретные линейки продуктов |

Настройте синонимы в Dashboard AACsearch в разделе Relevance Tuning → Synonyms. Поддерживается массовый импорт через CSV для больших наборов синонимов.

## Динамические правила ранжирования

Помимо весов полей, AACsearch поддерживает динамические правила ранжирования, которые корректируют релевантность на основе бизнес-логики:

```json
{
	"ranking_rules": [
		{
			"name": "Повысить товары в наличии",
			"condition": "in_stock :=: true",
			"boost": 1.5,
			"priority": 1
		},
		{
			"name": "Повысить новинки",
			"condition": "created_at:>=:2025-04-01",
			"boost": 1.2,
			"priority": 2
		},
		{
			"name": "Повысить продвигаемые товары",
			"condition": "is_promoted :=: true",
			"boost": 2.0,
			"priority": 0
		},
		{
			"name": "Понизить уценённые товары",
			"condition": "is_clearance :=: true",
			"boost": 0.5,
			"priority": 3
		}
	]
}
```

Эти правила применяются ко ВСЕМ поисковым запросам. Для boost-ов, специфичных для конкретного запроса, используйте параметры API `prioritize_fields` или `pinning`.

## Pinning и Burying

Иногда требуется хирургический контроль:

**Pinning (закрепление):** Принудительное размещение конкретного документа на позиции #1 для определённых запросов:

```json
{
	"pinning": {
		"query": "running shoes",
		"product_id": "featured_001",
		"position": 1
	}
}
```

**Burying (опускание):** Смещение нерелевантных результатов вниз:

```json
{
	"burying": {
		"query": "iphone",
		"conditions": [
			{ "field": "category", "value": "Accessories" }, // Чехлы и зарядки ниже телефонов
			{ "field": "condition", "value": "refurbished" } // Восстановленные внизу
		]
	}
}
```

## Цикл аналитической обратной связи

Релевантность — это не разовая настройка. Это непрерывный цикл улучшений, обеспечиваемый встроенной аналитикой AACsearch:

```
Сбор данных запросов → Анализ запросов без результатов → Добавление синонимов → Мониторинг CTR → Настройка весов
      ↕                                                                              ↕
Обновление правил ранжирования ← Выявление неэффективных поисков ← Обзор A/B-тестов ← Внедрение изменений
```

### Ключевые метрики аналитики

| Метрика            | Что она показывает                                            |
| ------------------ | ------------------------------------------------------------- |
| Zero-result rate   | Пробелы в контенте — пользователи ищут то, чего нет в индексе |
| Click-through rate | Правильные ли результаты в топе?                              |
| Query frequency    | Самые частые запросы — сфокусируйте усилия по настройке здесь |
| No-click rate      | Пользователи не нашли то, что искали, в топ-10 результатов    |
| Search-to-purchase | Конвертируют ли ищущие после нахождения результатов?          |
| Session depth      | Сколько поисков до нахождения? Высокое = плохая релевантность |

### Практическое руководство

1. **Неделя 1:** Проанализируйте запросы без результатов → добавьте недостающие товары или синонимы
2. **Неделя 2:** Проанализируйте топ-результаты с низким CTR → скорректируйте веса полей
3. **Неделя 3:** Добавьте синонимы для топ-запросов без результатов (обычно отраслевой жаргон)
4. **Неделя 4:** Проанализируйте сезонные тренды → добавьте/удалите правила pinning
5. **Постоянно:** Мониторьте еженедельный аналитический отчёт — 15 минут в неделю

## A/B-тестирование изменений релевантности

Каждое изменение релевантности должно быть измерено. AACsearch поддерживает A/B-тестирование:

```json
{
	"ab_test": {
		"name": "Увеличение веса brand vs description",
		"control": { "brand_boost": 2, "description_boost": 1 },
		"variant": { "brand_boost": 3, "description_boost": 1.5 },
		"traffic_split": 50,
		"metrics": ["ctr", "conversion", "zero_result_rate"],
		"duration_days": 7
	}
}
```

Dashboard покажет, какой вариант лучше по каждой метрике. Внедрите победителя.

## Модель зрелости релевантности

| Уровень | Характеристики                                              | Используемые инструменты              |
| ------- | ----------------------------------------------------------- | ------------------------------------- |
| L1      | Стандартное ранжирование, без настройки                     | Стандартный TF-IDF                    |
| L2      | Настроены веса полей                                        | Dashboard boosts                      |
| L3      | Группы синонимов для топ-20 неэффективных запросов          | Менеджер синонимов + аналитика        |
| L4      | Динамические правила ранжирования + сезонный pinning        | Ranking rules + pinning API           |
| L5      | A/B-тестирование, непрерывное улучшение на основе аналитики | Все инструменты + Dashboard аналитики |

Большинство команд достигают L3 в течение 2 недель после запуска. L4–L5 требуют ежемесячных вложений 2–4 часов.

## Распространённые ошибки в релевантности

| Ошибка                                      | Эффект                                                                | Исправление                                           |
| ------------------------------------------- | --------------------------------------------------------------------- | ----------------------------------------------------- |
| Слишком много поисковых полей               | Результаты совпадают по нерелевантным полям, низкий CTR               | Ограничьтесь 5 самыми важными полями                  |
| Все поля с одинаковым весом                 | Общие результаты — совпадение в названии не лучше совпадения в тексте | Используйте иерархию весов 3x/2x/1x                   |
| Отсутствие синонимов                        | Пользователи ищут «sneakers», получают «нет результатов»              | Добавьте отраслевые группы синонимов                  |
| Игнорирование аналитики нулевых результатов | Пробелы в контенте сохраняются месяцами                               | Еженедельный обзор, добавление контента или синонимов |
| Разовая настройка релевантности             | Релевантность ухудшается по мере роста каталога                       | Ежемесячный цикл настройки                            |
| Отсутствие A/B-тестирования                 | Изменения релевантности — это догадки                                 | Запускайте 50/50 тесты на 7 дней                      |
