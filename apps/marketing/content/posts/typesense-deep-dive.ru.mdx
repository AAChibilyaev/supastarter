Когда мы приступили к созданию AACSearch, мы оценили все основные поисковые движки на рынке. Вот почему победил Typesense, как мы спроектировали архитектуру вокруг него и как выглядит производительность в продакшене.

## Почему не Elasticsearch?

Elasticsearch — мощный, но тяжёлый в эксплуатации. Управляемый кластер Elasticsearch на Elastic Cloud начинается от 95 $/мес за 2 ГБ ОЗУ — до добавления каких-либо поисковых функций. Что более важно:

- **Мультиарендность сложна**: каждому арендатору нужен либо отдельный индекс (ресурсоёмко), либо сложная маршрутизация псевдонимов
- **Накладные расходы памяти**: JVM-куча Elasticsearch печально известна своей прожорливостью; кластер на 4 ГБ может реально проиндексировать только около 1 млн небольших документов
- **Поверхность конфигурации**: более 200 настроек создают DevOps-налог, который SaaS-команды не должны платить

Typesense, напротив, написан на C++ — он запускается за миллисекунды и использует лишь малую часть памяти.

## Почему не Meilisearch?

Meilisearch обладает отличным опытом разработчика, но на момент оценки:

- **Нет нативной мультиарендности** — каждому арендатору требуется отдельный узел или встроенная фильтрация токенов
- **Ориентирован на однопользовательские сценарии** (документация, блоги), а не на SaaS
- **Меньше вариантов развёртывания** для управляемого хостинга

Встроенная модель Typesense «коллекция на арендатора» идеально соответствует мультиарендности SaaS.

## Мультиарендная архитектура

```
┌─────────────┐     ┌──────────────┐     ┌─────────────┐
│  API Layer  │────▶│  Ingest      │────▶│  Typesense  │
│  (Hono/oRPC) │     │  Buffer      │     │  Cluster    │
└─────────────┘     │  (Postgres)  │     └─────────────┘
                    └──────────────┘
                           │
                    ┌──────┴──────┐
                    │  Worker      │
                    │  (bulk upsert)│
                    └─────────────┘
```

Каждый арендатор (организация) получает свою собственную коллекцию Typesense. Буфер приёма гарантирует, что мы никогда не перегрузим Typesense конкурентными записями — публичные API-запросы сначала попадают в Postgres, а фоновый worker выполняет непосредственный `bulkUpsert` в Typesense.

**Изоляция арендаторов**: вызовы содержат `tenantId: verified.organizationId`. API-токены с ограниченной областью действия дополнительно сужают доступ — `scopedFilter` комбинируется по И с фильтрами вызывающей стороны, что исключает чтение данных между организациями.

## Переиндексация без простоев

Это ключевая функция AACSearch. Когда вы обновляете свою схему:

1. Мы создаём новую коллекцию (v2) с обновлённой схемой
2. Мы передаём ваши документы из текущей коллекции (v1) в v2
3. Мы запускаем обе коллекции параллельно — v1 обслуживает трафик
4. Как только v2 полностью проиндексирована, мы атомарно меняем псевдоним
5. v1 удаляется

Весь процесс происходит без единого неудачного запроса.

## Бенчмарки производительности

Измерено на одном узле Typesense (8 ГБ ОЗУ, 4 vCPU):

| Тип запроса                         | p50  | p95  | p99   |
| ----------------------------------- | ---- | ---- | ----- |
| Простой поиск по ключевому слову    | 3ms  | 8ms  | 15ms  |
| Фасетный фильтр + поиск             | 5ms  | 12ms | 22ms  |
| Многополевой поиск (5 полей)        | 4ms  | 10ms | 18ms  |
| Поиск с геофильтром                 | 6ms  | 15ms | 28ms  |
| Семантический поиск (с эмбеддингом) | 45ms | 80ms | 120ms |

Эти бенчмарки получены на нашей хостинговой инфраструктуре (Docker под управлением Coolify). Задержка варьируется в зависимости от региона, но остаётся ниже 50 мс p99 для стандартного поиска в Северной Америке и Европе.

## Паттерны проектирования схем

Для большинства сценариев использования SaaS лучше всего работает плоская модель документов:

```typescript
interface Product {
	id: string;
	name: string; // полнотекстовый поиск
	description: string; // полнотекстовый поиск
	category: string; // фасет
	price: number; // фасет + сортировка
	tags: string[]; // фасет (множественный)
	inStock: boolean; // фильтр
	imageUrl: string; // только отображение результата
	metadata: Record<string, unknown>; // сквозная передача
}
```

Выводимая схема Typesense означает, что вам не нужно предварительно определять сопоставление. Для продвинутых сценариев вы можете задать веса для отдельных полей, пользовательские правила ранжирования и группы синонимов через панель управления AACSearch.

## Итог

Typesense предоставил нам поисковый движок, который:

- **Быстр**: среда выполнения C++, запросы менее 10 мс
- **Мультиаренден по дизайну**: модель «коллекция на организацию»
- **Прост**: выводимые схемы, минимальная конфигурация
- **Экономичен**: в 10 раз дешевле кластеров Elasticsearch аналогичного размера

Это был правильный фундамент для AACSearch.
