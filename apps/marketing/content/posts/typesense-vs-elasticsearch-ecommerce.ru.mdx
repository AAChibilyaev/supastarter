Два самых распространённых ответа на вопрос «какой поисковый движок использовать для e-commerce?» — Elasticsearch и Typesense. Оба являются отличным программным обеспечением, решающим схожие, но разные задачи. Неправильный выбор для вашего контекста стоит недель инженерного времени.

Вот прямое сравнение для задач e-commerce — и честная оценка того, когда вам вообще не стоит думать о выборе движка.

## Под что оптимизирован каждый движок

**Elasticsearch** был создан для агрегации логов и полнотекстовой аналитики в масштабе. Shopify, GitHub и Uber используют его для поиска по терабайтам логов. Его модель данных (инвертированный индекс + BKD-дерево для чисел), конвейер агрегаций и распределённая архитектура оптимизированы именно под такую нагрузку. Поиск по e-commerce — это то, что он умеет делать, а не то, для чего он был создан в первую очередь.

**Typesense** был разработан с нуля для интерактивного поиска. Вопрос, на который он должен был ответить: как возвращать релевантные результаты менее чем за 50мс для каталога из 10 миллионов товаров с устойчивостью к опечаткам, фасетной навигацией и настраиваемым ранжированием? Всё в его архитектуре — хранение поискового индекса в памяти, реализация на C++, schema-first модель коллекций — вытекает из этого ограничения.

Конкретно для поиска товаров в e-commerce Typesense выигрывает аргумент соответствия архитектуры по самому замыслу.

## Производительность

### Латентность запросов

| Движок        | Типичный p50 | Типичный p99 | При 1М документов |
| ------------- | ------------ | ------------ | ----------------- |
| Typesense     | 2–10мс       | 15–30мс      | 5–20мс            |
| Elasticsearch | 20–50мс      | 100–200мс    | 50–150мс          |

Эти цифры взяты из опубликованных бенчмарков на сопоставимом железе. Хранение индекса в памяти — основная причина: весь поисковый индекс живёт в RAM, поэтому дисковых операций ввода-вывода на горячем пути нет. Elasticsearch использует индекс на диске с кэшем страниц ОС; при росте нагрузки это вносит непредсказуемость в задержки.

Для виджета поиска по мере ввода, где каждое нажатие клавиши вызывает запрос, эта разница ощутима. Пользователи воспринимают запросы Typesense как мгновенные; запросы Elasticsearch — как быстрые (по-прежнему отлично, но измеримо медленнее).

### Пропускная способность индексации

Elasticsearch обеспечивает более высокую сырую пропускную способность для очень больших каталогов (10М+ документов). Его распределённая архитектура позволяет горизонтальное шардирование между узлами. Typesense по умолчанию работает на одном узле — при очень большом количестве документов вы упрётесь в ограничения памяти раньше, чем в ограничения пропускной способности.

Для типичных e-commerce каталогов (10К–2М товаров) ни один из движков не является узким местом. При 5М+ документов горизонтальное масштабирование Elasticsearch начинает иметь значение.

## Операционная сложность

Именно здесь сравнение становится болезненным для большинства команд.

**Elasticsearch** требует специализированной операционной экспертизы:

- Размерность кластера: мастер-узлы, узлы данных, координирующие узлы
- Управление шардами: количество шардов, ребалансировка после масштабирования
- Тюнинг JVM: размер heap, настройка GC
- Управление жизненным циклом индексов: уровни данных hot/warm/cold
- Снапшоты и восстановление: стратегия резервного копирования для многотерабайтных индексов
- Безопасность: X-Pack, ролевое управление доступом, настройка TLS

Грамотно эксплуатировать Elasticsearch — это работа на полную ставку. Команды, которые делают это плохо, сталкиваются с split-brain, сбоями OOM, медленными запросами из-за слишком больших шардов и лагом репликации, приводящим к устаревшим результатам.

**Typesense** несравненно проще:

- Один бинарный файл (или Docker-образ)
- Минимальная конфигурация: API-ключ, директория данных, порт
- Схема задаётся на уровне коллекции в JSON
- Резервное копирование — это просто копирование директории

Разработчик без опыта работы с Elasticsearch может запустить Typesense в продакшене за один день. Это преимущество, а не ограничение.

## Сравнение функций для E-Commerce

| Функция                          | Typesense                 | Elasticsearch                                                 |
| -------------------------------- | ------------------------- | ------------------------------------------------------------- |
| Устойчивость к опечаткам         | Нативная, настраиваемая   | Через fuzzy query (сложнее)                                   |
| Префиксный поиск                 | Нативный                  | Через edge n-gram (требует ручной настройки)                  |
| Фасетная навигация               | Нативная (facet_by)       | Через агрегации (мощнее, сложнее)                             |
| Геопоиск                         | Нативный                  | Нативный                                                      |
| Пользовательское ранжирование    | Нативное (sort_by)        | Через function score query                                    |
| Синонимы                         | Нативные                  | Через синонимный фильтр токенов                               |
| Курирование/закрепление          | Нативное                  | Не нативное (требует логики на стороне приложения)            |
| Ранжирование на основе ML        | Через Typesense Rules     | Через Elasticsearch Learning to Rank (LTR)                    |
| Индексация в реальном времени    | Нативная                  | Почти реальное время (интервал обновления 1 секунда)          |
| Многоязычный поиск               | 8 языков                  | Полный набор языковых анализаторов                            |

Для стандартного поиска товаров в e-commerce — совпадение по названию, фильтрация по категориям, сортировка по цене, фасетная навигация — Typesense покрывает всё необходимое с более простой конфигурацией. Более широкий набор функций Elasticsearch (вложенные агрегации, Learning to Rank, многоязычные анализаторы) становится актуальным для сложных сценариев: поиск на маркетплейсах с разветвлёнными иерархиями атрибутов, персонализация на основе ML или мультитенантные каталоги со сложным управлением доступом.

## Стоимость

### Self-Hosted

Typesense более эффективно использует память, чем Elasticsearch, для поиска по каталогам. Каталог из 1 миллиона товаров требует примерно:

- **Typesense**: 2–4 ГБ RAM (в зависимости от количества полей и длины текста)
- **Elasticsearch**: 8–16 ГБ RAM (overhead JVM + кэш страниц ОС + память сегментов)

По ценам облачных провайдеров Typesense может продуктивно работать на VPS за 20$/месяц. Elasticsearch, как правило, требует инстанса за 80–200$/месяц для аналогичного размера каталога.

### Управляемые сервисы

| Вариант       | Self-Hosted                              | Управляемый                                                         |
| ------------- | ---------------------------------------- | ------------------------------------------------------------------- |
| Typesense     | Бесплатно (open-source)                  | Typesense Cloud (по потреблению) или AACSearch                      |
| Elasticsearch | Бесплатно (open-source, базовые функции) | Elastic Cloud (от 95$/мес.), AWS OpenSearch (от 50$/мес.)           |

Цены Elastic Cloud для продакшн-нагрузки интернет-магазина начинаются от 150–400$/месяц. AACSearch (на базе Typesense) начинается от 29$/месяц за 1М поисковых запросов.

## Когда что выбирать

**Выбирайте Typesense (или AACSearch), если:**

- Вы строите поиск по товарам для e-commerce (основной сценарий, под который он проектировался)
- Вы хотите запустить поиск за дни, а не недели
- Вы работаете на PrestaShop или Bitrix со стандартным каталогом товаров
- Бюджет имеет значение — вы не можете обосновать 400+$/месяц за Elastic Cloud
- Ваша команда не имеет специализированной экспертизы в поисковой инфраструктуре

**Выбирайте Elasticsearch, если:**

- Вам нужно искать по терабайтам логов или потокам событий (его исходный сценарий)
- Вам нужны сложные вложенные агрегации (например, поиск на маркетплейсе с глубоко вложенными атрибутами)
- Вам нужно ранжирование на основе ML (плагин Learning to Rank)
- Ваша команда уже имеет экспертизу в Elasticsearch
- Вы работаете на платформе, где Elasticsearch уже является частью инфраструктуры (например, ELK-стек для observability)

## Управляемый вариант: почему выбор движка отходит на второй план

Вот что мы видим чаще всего: команды тратят недели на сравнение Typesense и Elasticsearch, разворачивают инфраструктуру, пишут скрипты синхронизации, а затем ещё месяц занимаются её поддержкой. Через полгода они вложили 200+ инженерных часов в поисковую инфраструктуру вместо продуктовых функций.

Конкретно для e-commerce выбор движка значит куда меньше, чем операционная модель. AACSearch работает на Typesense «под капотом» и управляет всем: подготовка кластера, миграции схемы, переиндексация, синхронизация из вашей CMS, мониторинг и поисковый виджет. Вы получаете производительность и простоту Typesense, не обслуживая никакой инфраструктуры.

Единственная причина для самостоятельного хостинга любого из движков — если у вас есть требования, которые управляемая платформа не может выполнить: чрезвычайно большое количество документов, сложная логика пользовательского ранжирования или требования к хранению данных на собственных серверах.

Для подавляющего большинства магазинов на PrestaShop и Bitrix правильный ответ — не «какой движок», а «управляемый Typesense с нативными коннекторами к CMS».

Именно это мы и создали.
