Главная концепция должна быть такой:

**AACSearch OS — не “поиск как сервис”, а премиальная операционная система поиска и discovery для продуктов, каталогов, документации и внутренних знаний.**

То есть продукт должен восприниматься не как очередной Algolia-alternative, а как **control plane для всего поискового слоя**: text search, voice search, image search, vector search, recommendations, suggestions, GraphRAG, analytics, relevance, security, tenants, quotas.

Apple в HIG прямо связывает типографику с читаемостью, иерархией, важностью контента и выражением бренда; для AACSearch OS это критично, потому что “дорогой” интерфейс создаётся не декором, а точной иерархией, spacing, шрифтом и спокойными состояниями. ([Apple Developer][1]) Material Design формулирует layout как инструмент направления внимания и облегчения действия; значит, страница должна вести пользователя через систему, а не просто перечислять фичи. ([Material Design][2])

# 1. Главная проблема текущей страницы

Сейчас страница уже стала лучше: есть AACsearch, Search Copilot, p99, SLA, security, pricing, analytics. Но она всё ещё выглядит как **очень насыщенный search SaaS landing**, а не как **дорогая инфраструктурная платформа**.

Что мешает воспринимать продукт серьёзно:

```txt
1. Hero всё ещё говорит “поиск, который понимает текст, голос и изображения”.
   Это звучит как AI-фича, а не как Search OS.

2. В DOM Search Copilot рендерится дважды.
   Это создаёт ощущение технической неаккуратности.

3. На странице слишком много секций одинаковой визуальной силы.
   Нет ощущения архитектуры: Input → Understanding → Retrieval → Intelligence → Control → Security.

4. Suggestions, recommendations, vector search, image search и GraphRAG не вынесены как стратегические слои.
   Они должны быть частью концепции AACSearch OS.

5. Algolia comparison слишком агрессивный и тяжёлый для главной.
   Для premium-стиля лучше говорить “predictable unit economics”, а не “мы дешевле в 50x”.

6. Footer всё ещё содержит “acme” и “Поиск как сервис”.
   Это ломает новый уровень позиционирования.

7. Много текста объясняет фичи, но мало текста объясняет систему.
   Search OS должен продаваться как operating layer, а не как список capabilities.
```

Новая цель:

```txt
Сайт должен ощущаться как Stripe / Linear / Vercel / Retool / Datadog-level infrastructure product:
точный, спокойный, дорогой, строгий, инженерный, но понятный бизнесу.
```

# 2. Основное позиционирование AACSearch OS

## One-liner

```txt
AACSearch OS — операционная система поиска и discovery для современных продуктов.
```

## Expanded version

```txt
Единый control plane для text, voice, image и vector search, recommendations, suggestions, GraphRAG, релевантности, аналитики и безопасного доступа.
```

## Более продуктовая версия

```txt
AACSearch OS превращает поиск из отдельной фичи в управляемый продуктовый слой: пользователь ищет текстом, голосом, изображением или вопросом, а система возвращает результаты, подсказки, рекомендации или ответ с источниками.
```

## Что значит “OS”

Слово **OS** должно быть доказано структурой сайта. Нужно показывать не “много функций”, а **слои операционной системы**:

```txt
Input Layer
Understanding Layer
Retrieval Core
Intelligence Layer
Knowledge Graph / GraphRAG
Experience Layer
Control Plane
Security & Governance
Analytics
```

GraphRAG нужно показывать как отдельный слой для задач, где важны связи между сущностями. Microsoft GraphRAG описывает его как структурированный, иерархический подход к RAG, который строит knowledge graph из данных и использует эти структуры для RAG-задач, а не просто ищет по plain text snippets. ([Microsoft на GitHub][3])

# 9. Knowledge Graph / GraphRAG

GraphRAG должен быть одной из самых дорогих секций сайта.

```txt
GraphRAG для вопросов, где важны связи
```

## Subtitle

```txt
Обычный RAG ищет похожие фрагменты. GraphRAG использует связи между сущностями: документами, товарами, пользователями, ошибками, инцидентами, сервисами и владельцами.
```

## Примеры запросов

```txt
Какие изменения в billing повлияли на Enterprise-клиентов?
Какие инциденты связаны с checkout?
Какие документы объясняют error 401 в SDK?
Какие товары похожи по атрибутам и поведению покупателей?
Кто владелец сервиса, связанного с последним инцидентом?
```

## Карточки

### Entity extraction

```txt
AACSearch OS выделяет сущности: users, organizations, errors, docs, incidents, SKUs, policies и services.
```

### Relationship graph

```txt
Строит связи: owns, depends on, caused by, similar to, mentioned in, purchased with.
```

### Graph retrieval

```txt
Находит не только похожий текст, но и релевантные связи вокруг запроса.
```

### Answers with citations

```txt
Ответ строится поверх источников и графа связей, чтобы пользователь мог проверить контекст.
```

## Визуальный стиль

Тонкая graph-диаграмма:

```txt
Service: billing
  → incidents
  → owners
  → docs
  → API methods
  → customers affected
```

Не делать яркую network-visualization. Она должна быть спокойной: тонкие линии, маленькие узлы, 2–3 акцентных связи.

# 16. Светлая тема

Светлая тема должна быть **cold premium**, не бело-серая.

## Палитра

```txt
Page background: #F7FAFF
Surface: #FFFFFF
Surface soft: #F8FBFF
Surface muted: #F1F5FA
Border subtle: rgba(15, 23, 42, 0.08)
Border default: rgba(15, 23, 42, 0.12)
Text primary: #0B1220
Text secondary: #334155
Text muted: #64748B
Blue accent: #2563EB
Cyan accent: #0891B2
Violet accent: #6D28D9
Emerald trust: #047857
Amber warning: #B45309
```

## Правило

```txt
Белый использовать только как surface.
Фон страницы — слегка холодный.
Серые border заменить на slate/blue-tinted border.
Акцентные цвета использовать редко.
```

Material Design подчёркивает, что color system должен поддерживать иерархию, состояния и бренд, а не просто украшать интерфейс. ([Material Design][4]) Для AACSearch OS это значит: blue/cyan/violet используются как смысловые роли, а не как градиенты ради красоты.

# 19. Отступы и сетка

Использовать 4/8px scale.

```txt
Page container: 1180px
Wide container: 1320px
Narrow text container: 760–860px

Desktop section padding: 104–136px
Compact section: 72–96px
Mobile section padding: 56–72px

Card padding desktop: 28–36px
Card padding mobile: 20–24px

Grid gap desktop: 24–32px
Grid gap mobile: 16px

Hero gap desktop: 64–88px
Section title → content: 40–56px
```

На mobile нельзя просто “сжать desktop”. Нужно менять композицию: таблицы → stacked cards, диаграммы → vertical flow, dashboard → compact mockup. WCAG 2.2 также требует, чтобы контент мог reflow без потери информации и функциональности на ширине 320 CSS px, а обычный текст имел контраст минимум 4.5:1; для UI-компонентов и значимых графических объектов нужен минимум 3:1. ([W3C][5])

# 25. Definition of Done

Страница готова к уровню “дорого, серьёзно, продуманно”, если выполняется:

```txt
1. Везде используется AACSearch OS, а не “поиск как сервис”.
2. Hero объясняет платформу, а не одну multimodal-фичу.
3. Есть OS Stack: Input → Understanding → Retrieval → Intelligence → Control → Security.
4. Image search, voice search, vector search, recommendations, suggestions и GraphRAG показаны как слои системы.
5. Search Copilot не дублируется в DOM.
6. Нет acme в footer, токенах и tenant examples, кроме намеренных demo-сцен.
7. Algolia comparison не доминирует на главной.
8. Светлая тема cold premium, тёмная тема graphite, без кислотности.
9. Типографика Onest / CoFo-level, body не тоньше 400.
10. Отступы крупные, карточки не плотные.
11. Все таблицы и diagrams имеют mobile-specific layout.
12. Контраст проходит WCAG AA.
13. Icon-only controls имеют aria-label и focus-visible.
14. Все product visuals выглядят как реальные интерфейсы, а не AI-art.
15. Финальный CTA говорит “Запустите AACSearch OS”, а не просто “создать аккаунт”.
```

AACSearch OS — премиальная поисковая операционная система для product-grade discovery:
text, voice, image, vector search, recommendations, suggestions, GraphRAG,
