# 1. Главная концепция AACSearch OS

**AACSearch OS — это операционная система поиска для современных продуктов.**

Не просто API. Не просто индекс. Не просто vector search. А единый слой, который управляет:

```txt
текстовым поиском
voice search
image / photo search
vector search
semantic search
autocomplete
suggestions
recommendations
GraphRAG
AI answers
ranking
facets
analytics
security
multi-tenant access
```

Главная формула:

```txt
AACSearch OS = Search + Discovery + Recommendations + GraphRAG + Control Plane
```

Или более продуктово:

```txt
AACSearch OS превращает поиск из отдельной фичи в управляемый продуктовый слой:
пользователь ищет, система понимает намерение, находит сущности, предлагает следующее действие,
строит рекомендации, отвечает с источниками и показывает команде, что улучшать.
```

# 2. Что значит “OS” в названии

Слово **OS** должно быть оправдано. Нельзя просто назвать обычный search API “Search OS”. Нужно показать, что это система с несколькими слоями.

## AACSearch OS состоит из 7 слоёв

```txt
1. Input Layer
2. Understanding Layer
3. Retrieval Layer
4. Intelligence Layer
5. Experience Layer
6. Control Plane
7. Security & Governance Layer
```

Это и есть концептуальная основа всего сайта.

# 4. Understanding Layer — слой понимания

Это ключевой слой, который делает продукт сильнее обычного Typesense/Algolia wrapper.

AACSearch OS должен не просто искать строку. Он должен нормализовать и понимать запрос.

## Что делает слой понимания

```txt
transcription для voice search
OCR / visual captioning для screenshot search
image embeddings для photo similarity
text embeddings для semantic search
query normalization
typo correction
language detection
intent detection
entity extraction
synonyms
filters inference
```

OpenAI описывает embeddings как способ превращать текст в числа для сценариев поиска и кластеризации; semantic search поверх vector stores позволяет находить релевантные результаты даже без точного совпадения слов в запросе. ([OpenAI Developers][2])

## Как это позиционировать

```txt
AACSearch OS сначала понимает намерение, затем выбирает правильный retrieval path:
keyword, vector, image similarity, graph traversal, recommendation или AI answer.
```

Это очень важно: **Search OS не всегда делает один и тот же поиск**. Он выбирает маршрут.

# 5. Retrieval Layer — ядро поиска

Это то, что физически находит результаты.

## Внутри Retrieval Layer

```txt
Full-text search
Prefix search
Typo-tolerant search
Faceted search
Vector search
Image similarity search
Hybrid search
Graph retrieval
Multi-search
Federated search
```

Typesense поддерживает vector search через embeddings и nearest-neighbor/KNN поиск; в документации также указано, что vector search может использоваться для similarity search, recommendations, semantic search и visual search. ([typesense.org][3])

Для AACSearch OS это нужно объяснять не как “у нас есть vector search”, а как:

```txt
Один retrieval layer для точных совпадений, смысловой близости, визуального сходства и связей между сущностями.
```

## Внутри Intelligence Layer

```txt

Recent searches
Pinned suggestions
Typo recovery
No-result recovery
Related results
Similar items
Personalized recommendations
Trending recommendations
GraphRAG answers

```

Typesense описывает рекомендации как сценарий, построенный на действиях пользователя в сессии и vector search; это хорошо ложится в AACSearch OS как recommendation layer, а не отдельная фича. ([typesense.org][4])

## Главная идея

Пользователь не всегда знает, что искать. AACSearch OS должен:

```txt
понять намерение
показать подсказку
исправить запрос
предложить похожий объект
показать рекомендации
дать ответ с цитатами
сохранить аналитику для команды
```

# 7. GraphRAG Layer — отдельный сильный модуль

GraphRAG нельзя прятать в “AI answers”. Это отдельная большая ценность.

## Что такое GraphRAG в AACSearch OS

```txt
GraphRAG строит слой связей между документами, сущностями, людьми, товарами, ошибками, темами и событиями.
```

Обычный RAG отвечает по похожим чанкам. GraphRAG должен отвечать по связям.

## Примеры

Для SaaS:

```txt
пользователь → организация → тариф → invoices → incidents → support tickets
```

Для документации:

```txt
API method → parameter → error code → migration guide → SDK function
```

Для e-commerce:

```txt
товар → бренд → категория → атрибуты → похожие товары → покупательские сессии
```

Для knowledge base:

```txt
policy → owner → department → related incidents → runbook
```

## Как это назвать в продукте

Лучшие названия:

```txt
Knowledge Graph
GraphRAG Layer
Entity Graph
Relationship Search
Context Graph
```

Я бы использовал:

```txt
Knowledge Graph + GraphRAG
```

Потому что “GraphRAG” звучит технически, а “Knowledge Graph” понятно для бизнеса.

## Текст для сайта

```txt
GraphRAG отвечает не только по похожим фрагментам текста, но и по связям между сущностями.
AACSearch OS строит knowledge graph поверх документов, товаров, пользователей, ошибок и событий,
чтобы отвечать на вопросы, где важен контекст.
```

```txt
а команде даёт control plane для релевантности, аналитики, безопасности и GraphRAG.
```

# 26. Итоговая идея

AACSearch OS должен восприниматься так:

```txt
Это не поисковик.
Это инфраструктура discovery внутри продукта.

Он принимает любые формы запроса:
текст, голос, фото, изображение, скриншот, диалог.

Он понимает намерение:
опечатки, смысл, entity, контекст, связи.

Он ищет разными способами:
keyword, vector, visual, graph, hybrid.

Он помогает пользователю:
autocomplete, suggestions, recommendations, no-result recovery, AI answers.

Он помогает команде:
analytics, relevance control, A/B tests, security, tenants, quotas.

Он масштабируется как платформа:
один API, один индексный слой, один control plane, один security model.
```

Это и есть сильная концепция **AACSearch OS**: не “ещё один Algolia alternative”, а **операционный слой для поиска, рекомендаций и GraphRAG в современных продуктах**.
